## 一、产品是什么

**知舟**：一个开源的个人知识管理工具。用户把链接/文件丢进来 → AI 自动摘要、标签、分类 → 用户审核确认 → 存入知识库 → 随时搜索召回。

**核心约束**：平台不提供 AI。用户自己绑 LLM API Key（DeepSeek / 混元 / OpenAI），所有 AI 处理走用户账户。

**版本体系**：分免费版（开源）和专业版（订阅），两个版本同步迭代。免费版含全部核心功能 + 10GB 云端或本地存储；专业版面向开发者，增加 MCP Server、开放 API 等高级功能。本地部署默认解锁全部功能。版本控制仅在官方云端服务层面生效，开源代码不区分版本。

## 二、产品理念（写进 README）

- **解放和发展生产力**：AI 数据爆炸时代，人的脑容量无法处理海量信息。知舟让各行各业用上 AI，把人从繁琐信息中解放。
- **顺应国家政策**：自主可控、开源开放，立足中国内容生态，助力中国式现代化。

## 四、技术栈

| 层 | 技术 | Go 依赖包 |
|----|------|----------|
| 前端 | React 18 + TypeScript + Vite 5 | — |
| 后端 | Go 1.22 | `gin-gonic/gin` `google/uuid` `lib/pq` `pgvector-go` `golang-jwt/jwt/v5` `golang.org/x/crypto` `go-readability/v2` |
| 数据库 | PostgreSQL + pgvector | — |
| 向量搜索 | pgvector | `github.com/pgvector/pgvector-go` |
| 网页解析 | go-readability v2 | `codeberg.org/readeck/go-readability/v2`（原 go-shiori 版已于 2025-12 归档） |
| LLM | 用户自配 API Key，兼容 OpenAI 接口格式 | — |
| 认证 | 手机号 + 短信验证码（JWT Bearer Token） | `golang-jwt/jwt/v5` |
| 加密 | AES-256-GCM（保护用户 API Key） | `crypto/aes` + `crypto/cipher`（标准库） |
| 部署 | Linux + Docker，提供一键部署脚本 | — |

> ⚠️ **Embedding 维度不是固定的**：DeepSeek 兼容 OpenAI 的 1536 维，但混元为 1024、豆包为 4096。数据库 `VECTOR(N)` 的 N 应通过环境变量 `EMBEDDING_DIM` 控制，不应硬编码。

> ⚠️ **go-readability v2 API 与 v1 不兼容**：v2 中 `article.Title` 变为方法 `article.Title()`，`article.TextContent` 字段已删除，需用 `article.RenderText(&buf)` 获取文本。见「知舟-代码骨架参考.md」第 11 节。

> ⚠️ **PostgreSQL TEXT[] 扫描陷阱**：使用 `lib/pq` 驱动时，`rows.Scan` 不能直接将 `TEXT[]` 列扫入 `[]string`，必须用 `pq.Array(&tags)` 包装。这是国内 Go + PostgreSQL 项目最常见的运行时 bug 之一。

## 五、数据库设计

### 5.1 用户表 `users`

```sql
CREATE TABLE users (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    phone VARCHAR(20) UNIQUE NOT NULL,
    password_hash VARCHAR(255),           -- 可选，验证码登录时为空
    tier VARCHAR(20) DEFAULT 'free',      -- 'free' | 'pro'
    storage_mode VARCHAR(20) DEFAULT 'cloud', -- 'cloud' | 'local'
    storage_used BIGINT DEFAULT 0,        -- 已用存储字节数
    pro_expires_at TIMESTAMP,             -- 专业版到期时间，NULL = 免费版
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);
```

### 5.2 API Key 配置表 `api_keys`

```sql
CREATE TABLE api_keys (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID REFERENCES users(id),
    provider VARCHAR(50) NOT NULL,        -- 'deepseek', 'openai', 'hunyuan', 'custom'
    api_key_encrypted TEXT NOT NULL,       -- AES 加密存储
    base_url VARCHAR(255),                 -- 自定义 endpoint
    model VARCHAR(100),                    -- 'deepseek-chat', 'gpt-4o' 等
    is_active BOOLEAN DEFAULT true,
    created_at TIMESTAMP DEFAULT NOW()
);
```

### 5.3 内容表 `contents`

```sql
CREATE TABLE contents (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID REFERENCES users(id),
    
    -- 原始信息
    url TEXT,                              -- 原始链接
    title TEXT NOT NULL,
    source_type VARCHAR(50),               -- 'web', 'bilibili', 'github', 'file', 'manual'
    raw_content TEXT,                      -- 原文全文
    
    -- AI 生成
    summary TEXT,                          -- AI 摘要
    category VARCHAR(100),                 -- AI 分类
    tags TEXT[],                           -- AI 标签，PostgreSQL 数组
    
    -- 状态
    status VARCHAR(20) DEFAULT 'pending',  -- 'pending' | 'approved' | 'skipped'
    
    -- 向量
    -- 向量维度因模型而异，应通过环境变量 EMBEDDING_DIM 控制
    -- DeepSeek/OpenAI: 1536 | 混元: 1024 | 豆包: 4096
    embedding VECTOR(1536),                -- pgvector，用于语义搜索
    
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_contents_user_id ON contents(user_id);
CREATE INDEX idx_contents_status ON contents(status);
CREATE INDEX idx_contents_tags ON contents USING GIN(tags);
CREATE INDEX idx_contents_embedding ON contents USING ivfflat (embedding vector_cosine_ops);
```

### 5.4 分类表 `categories`

```sql
CREATE TABLE categories (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID REFERENCES users(id),
    name VARCHAR(100) NOT NULL,            -- 如 "后端/微服务"
    parent_id UUID REFERENCES categories(id),
    created_at TIMESTAMP DEFAULT NOW(),
    UNIQUE(user_id, name)
);
```

### 5.5 订阅表 `subscriptions`

```sql
CREATE TABLE subscriptions (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID REFERENCES users(id),
    tier VARCHAR(20) NOT NULL,            -- 'pro'
    plan VARCHAR(20) NOT NULL,            -- 'monthly' | 'yearly'
    status VARCHAR(20) DEFAULT 'active',  -- 'active' | 'expired' | 'cancelled'
    started_at TIMESTAMP NOT NULL,
    expires_at TIMESTAMP NOT NULL,
    cancelled_at TIMESTAMP,               -- 取消时间，NULL = 未取消
    created_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_subscriptions_user_id ON subscriptions(user_id);
CREATE INDEX idx_subscriptions_status ON subscriptions(status);
```

                "storage_limit": 10737418240, "pro_expires_at": "..."|"null",
                "features": ["mcp","api","..."] }

## 八、核心业务逻辑

### 8.1 内容采集流程（ingest）

```
用户贴链接
    ↓
Go 后端接收 → 存入 contents (status=pending)
    ↓
异步任务：
  1. go-readability 抓取网页正文
  2. 调用用户配置的 LLM API：
     - 生成 200 字摘要
     - 判断分类（从已有分类中选择或建议新分类）
     - 提取 3-5 个标签
  3. 生成 embedding 存入 pgvector
  4. 更新 contents: status='pending', summary, category, tags, embedding
    ↓
用户审核页看到新内容 → 确认/修改/跳过
```

### 8.2 LLM Prompt 设计

**摘要 Prompt**：
```
你是一个知识管理助手。请用 200 字以内总结以下文章的核心内容：
标题：{title}
正文：{content}
```

**分类 Prompt**：
```
用户已有的分类：{categories}
请判断以下文章属于哪个分类。如果匹配已有分类，返回分类名。
如果不匹配，建议一个新分类名（格式如 "技术/后端"）。
标题：{title}
摘要：{summary}
```

**标签 Prompt**：
```
从以下文章提取 3-5 个标签。标签为简短关键词如 "Go" "K8s" "微服务" "开源"。
返回 JSON 数组格式。
标题：{title}
摘要：{summary}
```

### 8.3 搜索逻辑

```
用户输入 "微服务框架"
    ↓
1. PostgreSQL 全文搜索：title + summary, ILIKE '%微服务%'
2. pgvector 语义搜索：embedding <=> query_embedding
3. 合并结果，去重，按相关度排序
4. 支持标签筛选叠加
```

### 11.5 后端分层架构示例

**API Key 模块**完整打通了 Model → Repository → Service → Handler → 路由注册四层。每层都采用接口+依赖注入。关键设计决策：

- Repository 层：`*sql.DB` 注入，参数化查询防 SQL 注入，`defer rows.Close()` 防连接泄漏
- Service 层：调用加密工具自动加密 API Key 后再存储
- Handler 层：使用 `c.GetString("userID")` 从中间件注入的 context 取用户身份
- 路由：`r.POST("/api/keys", h.Save)`

完整代码见「知舟-代码骨架参考.md」第 5-8 节。

### 11.6 内容采集闭环

- **采集服务**：`server/internal/service/ingest_service.go`，串联解析 → 入库
- **前端表单**：`web/src/features/ingest/IngestForm.tsx`，loading/error 状态管理

> ⚠️ **待改造为异步**：当前采集是同步执行。Phase 2 需引入 BullMQ + Redis 做异步任务队列，用户提交后立即返回 `status: "processing"`，Worker 后台消化。

## 十二、给 AI Agent 的构建指令

1. 先建项目骨架：`web/` 用 `create-react-app` 或 Vite，`server/` 用 `go mod init`
2. 先写数据库迁移文件，跑通 PostgreSQL + pgvector
3. 后端先做认证模块（手机验证码），再做内容 CRUD
4. LLM 调用抽象成 interface，支持 OpenAI 兼容格式
5. 前端先做登录和采集页，审核页是核心交互重点
6. 搜索最后做，先用 ILIKE，再加 pgvector
7. 每完成一个模块写测试
8. 版本控制：所有功能通过 Feature Flag 控制，代码库不拆分。中间件 `middleware/tier.go` 校验功能权限，本地部署（`DEPLOY_MODE=local`）默认跳过所有权限检查
9. 订阅模块：先做数据模型（`users.tier` + `subscriptions` 表），再做中间件鉴权，最后做前端定价页和支付对接
10. 存储配额：免费版云端 10GB 软限制，在 `POST /api/contents/ingest` 入口检查 `storage_used`，超限返回 `429 StorageQuotaExceeded`
