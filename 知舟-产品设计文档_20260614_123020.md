## 一、为什么需要「知舟」

### 1.1 场景素描

你每天在多个平台发现好内容：B站教程、GitHub 项目、CSDN 文章、公众号推送、群聊截图……随手点收藏，半年后想找时翻烂了也找不到。

不只是你。500 万+ 人用 Recall、Cubox 年营收千万级、MindPocket 开源即爆——说明「收藏了但用不上」是普世痛点。

### 1.2 现有产品的缺口

| 产品 | 擅长 | 短板 |
|------|------|------|
| Cubox | 稍后阅读 + AI 摘要 | 不支持视频/截图，面向阅读场景 |
| Recall | 知识图谱 + 间隔重复 | 偏西文生态，无中国平台深度集成 |
| Atlasnote | 全格式 + 本地优先 | 桌面端为主，无移动端 OCR |
| MindPocket | 开源 + B站字幕提取 | 需自部署，非开箱即用 |

**共同缺失的**：一个真正覆盖中国内容生态（B站、公众号、CSDN、掘金、Gitee、截图）的「从采集到消化到召回」全链路工具。

## 二、产品定义

### 2.1 一句话

> **知舟** — 任何内容丢进来，AI 读懂、归好类、你需要时一秒找到。平台不提供 AI，用户自带 API Key。

### 2.2 产品理念

**一、解放和发展生产力**

AI 数据大爆炸时代，人的脑容量无法处理海量信息。这些信息需要被整理、归纳、分类、存储——但不应由人亲自来做。知舟的使命是让各行各业用上 AI，把繁琐的信息处理交给机器，人只做决策和创造。不再被碎片信息困扰，不再为"收藏了找不到"焦虑。

**二、顺应国家政策，为中国式现代化贡献力量**

知舟坚持自主可控、开源开放的技术路线。立足中国内容生态，服务中国用户，以数字化工具助力新质生产力发展，为中国式现代化添砖加瓦。

### 2.3 核心管道

```
采集（任意入口） → 消化（AI 自动处理） → 审核（你扫一眼） → 归档（知识库）
                                                                  ↓
                                                         随时搜索 / 回顾
```

### 2.4 核心理念

知舟是一个纯客户端工具。**平台不提供任何 AI 能力**，所有摘要、标签、对话、转录功能均通过用户自己配置的 LLM API Key 驱动。用户绑定 DeepSeek、混元、OpenAI 等任意兼容 API 即可工作。

这意味着：
- 用户对自己的 AI 用量和费用完全可控
- 数据不出用户控制范围（API 调用直接走用户账户）
- 平台零 AI 成本，核心功能全部免费开源

### 2.5 核心设计原则

**反回音壁原则（Anti-Echo-Chamber）**

所有 RAG 类工具都有一个隐蔽陷阱：AI 只能在你的收藏夹里找答案。你收藏了 3 篇伪科学的睡眠博客，问"我失眠怎么办"——AI 答得头头是道，但源头是不知道谁写的博客。

知舟必须做到：

- 🚫 **绝不把收藏夹内容当作权威答案**。对话时明确区分「来自你收藏的 XX 文章说」和「来自公开知识」，绝不让用户混淆二者的分量。
- ✅ **默认开启联网交叉验证**。当用户问知识性问题（健康、法律、金融等），AI 必须同时检索公开权威资料，标注信息来源的可靠性。
- ✅ **来源透明标注**。每个回答附带信息源链接和你最初收藏时的日期，用户可以自己判断时效和可信度。
- ✅ **"我问的是 AI 还是我的收藏夹？"** ——用户可以一键切换：只看收藏 / 只看公开 / 两者混合。

**其他原则**

| 原则 | 说明 |
|------|------|
| **采集不动原内容** | 原文永久保留，AI 摘要只是辅助，不能替代原文阅读 |
| **审核门不可跳过** | 所有入库内容必须过用户眼睛，除非用户主动开启自动通过 |
| **AI 自配** | 平台不提供 AI，用户绑定自己的 LLM API Key。DeepSeek / 混元 / OpenAI 等任意兼容 API 均可 |
| **格式无关** | 网页/视频/PDF/截图一律平等，统一处理管线 |

## 三、功能架构

### 3.1 采集层（一切内容的入口）

| 采集方式 | 说明 | 优先级 |
|----------|------|:---:|
| **浏览器扩展** | 一键保存当前页面文章/项目页，覆盖 Chrome / Edge / Firefox / Safari | P0 |
| **移动端分享** | 系统分享菜单接入，任何 App 的内容都能丢进来 | P0 |
| **剪贴板智能识别** | 打开知舟自动检测剪贴板链接，一键确认上传 | P0 |
| **链接粘贴** | 手动贴链接到知舟 | P0 |
| **视频平台深度集成** | 支持 B站、今日头条、抖音、快手等主流视频平台，粘贴链接 → 自动提取字幕/简介 → 结构化总结 | P0 |
| **截图 OCR** | 截图 → 自动识别文字 → 归入对应分类 | P1 |
| **本地文件** | 拖拽 PDF/Word/MD 直接入库 | P1 |
| **RSS/订阅监控** | 关注 GitHub 仓库、博客 RSS，自动采集新内容；用户可选择偏好平台（如 GitHub、Gitee、掘金等） | P2 |
| **微信生态** | 公众号文章转存（待评估可行性，微信生态闭环） | P2 |

### 3.2 消化层（AI 自动处理）

| 能力 | 说明 | 优先级 |
|------|------|:---:|
| **自动摘要** | 每篇内容入库后自动生成 200 字结构化摘要 | P0 |
| **自动分类** | AI 判读内容主题，自动归入对应分类（如"后端/微服务""AI/LLM"） | P0 |
| **自动标签** | 提取技术栈、领域、类型等标签（如 `Go` `K8s` `教程` `开源`） | P0 |
| **视频转文字** | B站/YouTube 视频 → 字幕提取 → 全文可检索 | P0 |
| **全文翻译** | 外文内容自动翻译为中文摘要（可选保留原文） | P1 |
| **关键观点提取** | 标注原文核心论断，用引用块展示 | P1 |
| **知识图谱** | 自动关联相关内容，展示"你还收藏过这个" | P2 |

### 3.3 审核层（你的确认门）

| 行为 | 说明 |
|------|------|
| **快速浏览** | 采集后推送通知「已处理 3 条新内容，要看看吗？」 |
| **确认归档** | 摘要、分类、标签预览，一键确认或手动调整 |
| **批量审核** | 多条待审内容列表，左滑确认 / 右滑跳过 |
| **自动规则** | 设定「来自 GitHub 的全部自动通过」等规则 |

### 3.4 召回层（需要时一秒找到）

| 能力 | 说明 | 优先级 |
|------|------|:---:|
| **全文搜索** | 搜标题 + 摘要 + 正文 + 字幕文本 | P0 |
| **语义搜索** | "之前收藏的那个微服务框架" → 即使标题不含"微服务"也能搜到 | P0 |
| **标签筛选** | 按标签、平台来源、时间范围过滤 | P0 |
| **知识图谱浏览** | 从一个收藏点出发，看到关联的其他收藏 | P2 |
| **定期回顾** | "本周收藏了 5 篇 AI 相关，需要重点看的是这篇" | P2 |
| **AI 对话** | 基于收藏内容对话："我收藏的那些 K8s 文章里推荐的网络插件有哪些？" | P1 |

### 3.5 导出与分享

| 能力 | 说明 |
|------|------|
| Markdown 导出 | 单篇或批量导出为 MD |
| 分享链接 | 生成可读链接分享给他人 |

### 4.3 关键设计决策

**0. MCP Server 内置于应用层**

知舟的知识库价值不只在知舟 App 内——用户在 Cursor/Trae/Claude Code 里写代码时也需要引用收藏。MCP（Model Context Protocol）是当前 AI 工具通用的外部数据接入协议，Cursor、Trae、Claude Code 均原生支持。

知舟内置 MCP Server，暴露两个工具：
- `search_zhizhou(query, limit)` → 全文 + 语义搜索
- `fetch_zhizhou(id)` → 返回收藏原文

用户在 Cursor 里配置一行 MCP 地址即可。AI 编程时自动从知舟拉取收藏的技术文章、开源项目总结，无需手动切换工具。

**1. 消化引擎独立为微服务**

采集量小时没问题，但 OCR 和视频转录耗时较长（分钟级），不适合阻塞请求。建议拆为异步任务队列（BullMQ + Redis），采集 → 入队 → Worker 处理 → 结果回写。

**2. 向量数据库选 pgvector 而非独立部署**

初期用户量不大时，pgvector 和 PostgreSQL 统一运维，降低复杂度。日活过万后再考虑 Milvus。

**3. 用户自配 API Key（唯一方式）**

平台不提供 AI 服务。用户首次使用需绑定自己的 LLM API Key（DeepSeek / 混元 / OpenAI 等），所有 AI 处理（摘要、标签、对话、转录）走用户自己的账户。这让平台可以做到零 AI 成本，核心功能免费开源。

## 五之附：部署方案

知舟提供三种部署方式，适应不同用户的需求和技术能力。

### 方案一：官方云端（推荐普通用户）

知舟提供官方云服务器，用户注册即用，无需任何技术操作。数据加密存储，定期备份。免费版用户享有 10GB 存储空间，专业版无容量限制。

**适宜**：非技术用户、个人使用者、不想折腾运维的人。

### 方案二：用户自购服务器

用户自行购买云服务器（阿里云/腾讯云/华为云等），知舟提供完整部署文档和一键脚本。自部署默认解锁专业版全部功能。

**适宜**：有技术背景的用户、对数据有合规要求的团队、想完全掌控服务器的开发者。

### 方案三：本地部署

用户在自己机器上部署（Linux），数据完全本地化，不外传。本地部署默认解锁专业版全部功能。

**适宜**：极客、隐私敏感用户、离线环境、公司内部部署。

## 七、版本与商业模式

### 7.1 两个版本，同步迭代

知舟分为**开源免费版**和**专业版**，两个版本同时推送更新迭代升级，不存在"不用专业版就限制你"的商业行为。专业版的存在唯一理由是：官方云端服务器成本需要覆盖。

| | 免费版（开源） | 专业版 |
|---|---|---|
| **定位** | 个人知识管理的基础工具 | 面向开发者和深度用户的高级工具 |
| **核心功能** | ✅ 全部基本功能：采集、消化、审核、归档、搜索 | ✅ 全部免费版功能 |
| **存储** | 🅰 官方云端 10GB 或 🅱 完全本地存储（二选一） | 官方云端无容量限制 |
| **MCP Server** | ❌ | ✅ 内置 MCP Server，Cursor / Trae / Claude Code 直接调用知舟知识库 |
| **API 开放接口** | ❌ | ✅ 开放 REST API，支持第三方工具对接 |
| **高级功能** | 基础功能完整可用 | 后续持续添加高级能力（自动化规则引擎、批量操作 API、Webhook 等） |
| **更新节奏** | 与专业版同步推送 | 与免费版同步推送 |
| **价格** | 永久免费 | 按月/年订阅，定价仅覆盖服务器成本 |
| **部署方式** | 本地部署 / Docker / 源码编译 | 同左 + 官方云端托管 |

### 7.2 设计原则

1. **不歧视免费用户**：免费版拥有完整的核心体验，不是阉割版。所有"知识管理"的基础能力——采集、摘要、审核、搜索——免费版全部具备，体验不打折。

2. **专业版 = 服务器成本 + 开发者工具**：专业版收费的原因只有一个——官方云服务器的带宽、存储、计算资源有真实成本。专业版面向的是需要云端大容量存储、需要 MCP/API 对接开发工具链的深度用户，这些功能本身消耗服务器资源。

3. **同步更新，不分先后**：每次版本迭代，免费版和专业版同时发布。不会出现"新功能先给专业版，免费版等下个季度"的情况。

4. **本地部署永远免费且完整**：选择自部署的用户（无论本地还是自购服务器），享有专业版全部功能。开源代码不区分版本，版本控制仅在官方云端服务层面生效。

5. **不做商业焦虑**：不弹窗催升级、不限制导出、不降低免费版速度。免费用户不会感受到"被商业策略逼迫"的体验。

### 7.3 用户群体画像

| 群体 | 推荐版本 | 原因 |
|---|---|---|
| 普通知识工作者 | 免费版（云端 10GB） | 日常收藏足够，不想折腾部署 |
| 隐私敏感用户 | 免费版（本地部署） | 数据不出本机，完全免费 |
| 开发者 / 极客 | 专业版 | 需要 MCP 对接 IDE、API 集成到自己的工具链 |
| 技术博主 / 内容创作者 | 专业版 | 大量收藏 + 需要云端大容量 + 自动化规则 |
| 团队 / 小公司 | 专业版（多用户云端） | 需要共享知识库和 API 对接（后续团队版） |

### 7.4 技术实现要点

- **代码库不拆分**：免费版和专业版共用同一套代码，通过配置开关（Feature Flag）控制功能可见性
- **版本判断逻辑**：本地部署默认解锁全部功能；官方云端根据用户订阅状态判断
- **数据不锁定**：免费版用户随时可导出全部数据，切换到本地部署即可获得专业版体验
- **订阅管理**：专业版订阅通过官方云端账户管理，不引入第三方支付 SDK，保持轻量
